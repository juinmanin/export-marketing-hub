# 2인 협업 작업 규칙서

이 문서는 `Export Marketing Hub` 프로젝트를 2명이 함께 작업할 때 쓰는 실무용 규칙서다.

## 1. 기본 원칙

- `main` 브랜치는 항상 실행 가능한 상태로 유지한다.
- 작업은 각자 브랜치에서 하고, 완료 후 `Pull Request`로 합친다.
- 한 번에 큰 변경보다 작은 단위로 자주 합친다.
- 초보자는 혼자 큰 구조를 바꾸지 않고, 작은 작업부터 맡는다.
- 실행이 안 되거나 충돌이 생기면 바로 혼자 오래 붙잡지 말고 공유한다.

## 2. 2인 역할 추천

### A. 오너 역할

- 프로젝트 구조 이해
- 브랜치 생성과 병합
- 배포 확인
- 구조 변경, 상태관리, 캐시 관련 수정
- 최종 리뷰

### B. 초보자 역할

- 문구 수정
- 라벨/버튼 텍스트 수정
- 샘플 데이터 추가
- 사용설명 문서 수정
- 결과 화면의 작은 UI 수정
- 단일 섹션 중심 개선

## 3. 처음 1~2주 운영 방식

- 처음에는 화면 공유 또는 같이 보면서 작업한다.
- 초보자는 먼저 `git pull`, `git branch`, `git add`, `git commit`, `git push` 흐름을 익힌다.
- 구조를 어느 정도 이해한 뒤에 각자 독립 작업으로 넘어간다.

## 4. 브랜치 규칙

- `main`: 안정 버전
- 작업 브랜치: `codex/...` 형식 사용

예시:

- `codex/ui-help-page`
- `codex/blog-output-format`
- `codex/pdp-copy-pack`
- `codex/korean-market-samples`

## 5. 작업 시작 전 체크

매번 작업 시작 전에 아래 순서를 지킨다.

1. 저장소 최신 상태 받기
2. 새 브랜치 만들기
3. 작업 범위를 한 문장으로 정리하기
4. 수정 파일 수를 최소화하기

## 6. 권장 작업 분담 예시

### 오너가 맡는 것

- `app.js`의 상태관리 변경
- 실행 오류 수정
- 캐시/빌드 번호 갱신
- 배포와 저장소 관리

### 초보자가 맡는 것

- 도움말 문서 보강
- 특정 결과 문안 수정
- 한국어 라벨 다듬기
- 샘플 산출물 추가
- 특정 카드 UI 정리

## 7. Pull Request 규칙

- 제목은 무엇을 바꿨는지 바로 알 수 있게 쓴다.
- 하나의 PR에는 하나의 주제만 넣는다.
- PR 올리기 전에 로컬에서 실행 확인을 한다.
- 스크린샷이 있는 수정은 스크린샷을 함께 남긴다.

좋은 예:

- `블로그 결과를 제목/본문/메타 분리 복사 형식으로 변경`
- `한국 시장 샘플 2종 추가`
- `상세페이지 CTA 문안 개선`

피해야 할 예:

- `수정`
- `이것저것 고침`
- `최종`

## 8. 커밋 규칙

- 커밋은 작고 명확하게 한다.
- 커밋 메시지는 결과 중심으로 쓴다.

예시:

- `Add Korean blog sample output`
- `Improve smartstore upload copy blocks`
- `Fix onboarding field persistence`

## 9. 충돌 방지 규칙

- `app.js`를 동시에 크게 수정하지 않는다.
- 같은 파일을 만져야 하면 먼저 말하고 시작한다.
- 큰 구조 변경 전에는 작업자를 한 명으로 정한다.
- 초보자가 `app.js`를 고칠 때는 가능하면 오너와 같이 본다.

## 10. 실행 확인 규칙

작업 후 최소 아래를 확인한다.

1. 앱이 열리는지
2. 수정한 화면이 실제로 보이는지
3. 버튼이 눌리는지
4. 결과 복사가 되는지
5. 콘솔 오류가 없는지

## 11. 현재 프로젝트 기준 주의점

- 이 프로젝트는 `app.js`에 로직이 많이 모여 있어서 충돌이 잘 난다.
- `styles.css`는 다른 작업이 남아 있을 수 있으니 함부로 덮어쓰지 않는다.
- 빌드 번호와 캐시 버전이 바뀌면 `index.html`, `sw.js`, 실행 스크립트를 같이 본다.
- 결과 화면은 `전체 복사`와 `플랫폼별 업로드 패키지`를 함께 확인한다.

## 12. 추천 협업 흐름

### 초보자 작업 예시

1. 최신 코드 받기
2. `codex/help-copy-update` 브랜치 생성
3. 도움말 문구만 수정
4. 실행 확인
5. 커밋
6. PR 생성
7. 오너 리뷰 후 병합

### 오너 작업 예시

1. 최신 코드 받기
2. `codex/pdp-output-logic` 브랜치 생성
3. 출력 로직 수정
4. 실행 확인
5. 커밋
6. PR 생성
7. 검토 후 병합

## 13. 하지 말아야 할 것

- `main`에서 바로 작업
- 서로 말 없이 같은 파일 대규모 수정
- 실행 안 되는 상태로 푸시
- 의미 없는 커밋 메시지 작성
- 초보자에게 구조 변경을 처음부터 단독으로 맡기기

## 14. 가장 추천하는 운영 방식

현재는 아래 방식이 가장 안전하다.

- 오너 1명 + 초보자 1명 구조로 운영
- 처음엔 같이 보며 진행
- 초보자는 작은 작업부터 시작
- 익숙해지면 각자 브랜치에서 작업
- 최종 병합은 오너가 책임지고 진행
